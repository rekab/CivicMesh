# Victron BMV-712 battery monitor (BLE) setup

Optional. How to connect a **Victron BMV-712** battery monitor to a CivicMesh
node over Bluetooth Low Energy so the node samples true battery **State of
Charge**, voltage, and current. Most nodes have no battery monitor; this is only
for nodes that do.

The node listens **passively** to the BMV's encrypted "Instant Readout" BLE
advertisements (~1/second). There is **no pairing and no GATT connection** — so
it doesn't conflict with the VictronConnect phone app, and it minimizes radio
contention on a Pi Zero 2W (where WiFi and Bluetooth share one radio). The
runtime, schema, and stats surfaces are described in
[`docs/telemetry.md`](telemetry.md) § "Battery monitor"; this document is the
hands-on bring-up.

> Other Victron Instant Readout devices (SmartShunt, BMV-702) use the same
> mechanism and library, but the values and field availability below are
> described for the BMV-712.

---

## What you need

- A BMV-712 (its BLE is built in — no dongle).
- The node's Pi, running `civicmesh-mesh`, within BLE range of the BMV.
- The VictronConnect app (phone/tablet) to read the BMV's MAC and encryption key.

You will collect **two** values: the BMV's **BLE MAC** and its per-device
**advertisement encryption key**.

---

## Step 1 — Enable Instant Readout and copy the key (VictronConnect)

The advertisement key is **per device** and is generated by the BMV. You read it
out once from the app.

1. Open VictronConnect, connect to the BMV.
2. Open **Settings** (the gear, top-right) → the **⋮** menu → **Product info**.
3. Find **"Instant readout via Bluetooth"** and turn it **on**.
4. On the same screen, open **Encryption data** and copy the **encryption key**
   (32 hex characters). This is the value that goes in `encryption_key`.
5. Note the **MAC address** shown on the same Product-info screen (format
   `AA:BB:CC:DD:EE:FF`). This is `mac`.

> Menu wording shifts between VictronConnect versions; if "Product info" isn't
> where described, look for "Instant readout" under the device's advanced/
> Bluetooth settings. The key is sometimes labelled "advertisement key."
>
> The key changes if the BMV is **factory-reset or swapped** — re-copy it then,
> or decryption silently fails (the node will log `power:no_adverts`).

---

## Step 2 — Bring up Bluetooth on the Pi (headless)

On a fresh headless Pi the adapter is often soft-blocked or the service is off.

```sh
sudo rfkill unblock bluetooth
sudo systemctl enable --now bluetooth
bluetoothctl scan on        # watch ~10s; your BMV's MAC should appear, then Ctrl-C
bluetoothctl show           # "Powered: yes" confirms the adapter is up
```

Seeing the BMV's MAC in `scan on` proves the adapter works and the BMV is in
range and advertising. You do **not** pair or connect — the node only listens.

---

## Step 3 — Install the BLE dependencies

The BLE library `victron-ble` is a **base dependency**, installed by a plain
`uv sync` (and by `civicmesh promote`, which runs `uv sync --frozen` on the
target). It was formerly behind a `[power]` extra, but `uv sync --frozen`
installs base deps only — not extras — so the extra left it absent on promoted
nodes. There is nothing extra to install:

```sh
uv sync
```

`bleak` + `pycryptodome` arrive transitively (via `meshcore` and `victron-ble`).
`victron-ble==0.9.3` is pinned exact (see the comment in `pyproject.toml` for
why). The wheels include aarch64 builds, so this works on an offline Pi Zero 2W
that has the wheels cached.

---

## Step 4 — Smoke-test the link (before touching CivicMesh)

Prove the MAC + key + library decode **before** enabling the sampler, so a
failure points at one layer, not the whole integration. From the repo root:

```sh
uv run python scripts/ble_smoke.py --mac AA:BB:CC:DD:EE:FF --key 0123…ef
```

On a node that already has its `config.toml` populated, the equivalent on-node
check is `civicmesh power-test` — same `BLESource` path, reads the MAC/key from
`[power_monitor]`, decodes one advert (default ~30s) and prints SoC/V/A/W, exit
0 on decode / non-zero on no-data (it works regardless of `enabled`, so it's the
pre-flip check). `ble_smoke.py` stays the verbose bench variant below.

It listens for ~30s and prints decoded readings. **Good** output ends with
`RESULT: OK` and SoC/voltage/current that match VictronConnect live. If it prints
`RESULT: NO DATA`, work through its troubleshooting hints (and the table below)
before continuing.

Library cross-check (bypasses CivicMesh entirely):

```sh
victron-ble read "AA:BB:CC:DD:EE:FF@0123…ef"   # should stream decoded JSON
```

---

## Step 5 — Configure and enable

In `config.toml`, fill in the `[power_monitor]` section (see
[`config.toml.example`](../config.toml.example) for the full commented block):

```toml
[power_monitor]
enabled = true
transport = "ble"
mac = "aa:bb:cc:dd:ee:ff"
encryption_key = "0123456789abcdef0123456789abcdef"
sample_interval_sec = 60      # try 15 while testing, restore 60 after
stale_after_sec = 180
```

`config.toml` now holds a secret (the key) — `chmod 600 config.toml`.

Restart the mesh process and watch its log for `power:` lines:

```sh
sudo systemctl restart civicmesh-mesh
journalctl -u civicmesh-mesh -f | grep -i power
```

Expect `power:scanner_started`, then steady samples, and **no**
`power:no_adverts` warning.

---

## Step 6 — Verify it's sampling

**Database** (one row per `sample_interval_sec`):

```sh
sqlite3 <db_path> \
  "SELECT datetime(ts,'unixepoch','localtime'),soc,voltage_mv,current_ma,power_w \
   FROM power_samples ORDER BY ts DESC LIMIT 5;"
```

Values should be sane and match the app; current is **signed** (negative =
discharging).

**DM `stats`** — DM the node `stats`; once a fresh sample exists the reply
includes a `bat` line, e.g. `bat 99.7% 13.2V -2.1A`. The line is omitted
entirely when no monitor is configured or the latest sample is stale (older than
`stale_after_sec`), so a node without a working BMV spends no airtime on it. When
shown, an individual field reads `?` if just that value was unavailable (e.g. SoC
before the shunt syncs).

**Web** — `curl -s http://<node>/api/stats | jq .system.power` shows the latest
scalars, `last_sample_age_s`, and the SoC/voltage 1h series. The stats page
renders Battery SoC and voltage tiles with sparklines.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `scan on` never shows the BMV | BT blocked / adapter down / out of range | `rfkill unblock bluetooth`; `systemctl start bluetooth`; move the Pi closer |
| Smoke test: `NO DATA`, no decode ever | Wrong MAC, or adapter not scanning | Re-check `mac`; confirm `bluetoothctl show` → Powered: yes |
| Smoke test sees nothing, but `victron-ble read` also fails | Wrong/rotated encryption key | Re-copy the key from VictronConnect (it changes on BMV reset/swap) |
| `power:no_adverts` in the log, no rows | Scanner runs but decodes nothing — wrong key/MAC, **or** a victron-ble whose `Scanner.callback` signature changed | Verify the key; then verify the override (below) |
| `power:deps_missing` | victron-ble/bleak (base deps) failed to import — broken install | `uv sync` (on prod, re-run `civicmesh promote`) |
| Samples flow, then `power:gap` / stale tile | BMV powered off, out of range, or radio contention | Restore power/range; on Pi Zero 2W see the contention note below |
| `power:scanner_stalled … restarting` | Stream went quiet long enough to trigger a scanner restart | Self-healing; if it loops, check BT/adapter health |

**Verify the callback signature** (the silent-no-data case). On the node, against
the version you actually installed:

```sh
uv run python -c "import inspect, victron_ble.scanner as s; print(inspect.signature(s.Scanner.callback))"
# expect: (self, ble_device, raw_data, advertisement)
```

If it differs, our `BLESource` override in `power_monitor.py` stops firing and
the source goes dark with no error. The module documents the low-level fallback
path (`detect_device_type` + `parser(key).parse(raw)` on a plain
`bleak.BleakScanner`); update `BLESource` to match the new version and re-pin.

**Pi Zero 2W radio contention.** WiFi (the captive-portal AP) and Bluetooth share
one radio. Passive advertisement listening is low-cost, but if you see AP drops
or web-latency spikes that correlate with scanning, the wired **VE.Direct serial**
path is the robust alternative — the `transport = "vedirect"` config value is the
reserved seam for a future `VEDirectSource` (not yet implemented).

---

## Swapping or resetting the BMV

The encryption key is per-device and regenerates on a factory reset. After a swap
or reset, repeat **Step 1** to copy the new MAC/key, update `[power_monitor]`,
and restart `civicmesh-mesh`.
